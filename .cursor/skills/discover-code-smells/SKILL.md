# 发现代码坏味道（只读）

## 适用范围

- **纳入范围**：可维护性层面的「坏味道」（预警信号）：魔法字面量、超长方法/类、过深分支、重复代码、职责混杂、错误与边界处理不足、耦合过紧与隐式依赖、敏感信息硬编码、注释或文档与实现脱节等。
- **排除范围**：不实施任何修复、重构、格式化或重命名。**禁止修改代码。** 若用户要求一并修复，应完成清单输出后说明：修复须另起对话或使用单独的「仅修复」流程，并仅处理用户确认的编号。
- **工作假设**：在人工智能辅助编程下，**命名**与常规 **Lint** 往往已较规范；应优先关注**结构层面**问题及其对**后续改动成本**与**缺陷风险**的影响。

## 工作流程

1. **明确范围**：若用户未指定路径，须询问待分析的文件或目录（对应课程材料中提示词里的「……」）。
2. **只读分析**：使用读取、搜索类工具；不得应用补丁或执行破坏性操作。
3. **按类别归纳**：使用下文「类型代码与发现规则」表；若用户仅要求某一类，**不得**在同一轮中扩展到其他类型。
4. **统一编号**：每一发现项使用 **类型代码-序号**（如 `MNS-1`），每一类型内序号从 1 起连续递增。
5. **空结果**：某类型经检视后无问题时，在该类型下保留一条子项，写明 **「未发现」**。

## 类型代码与发现规则

下列规则与课程文档中「🔍 典型提示词（仅发现）」一致，须严格遵守各类**条数上限**与**范围边界**。

| 代码 | 问题类型 | 最多条数 | 发现要点 |
|------|----------|----------|----------|
| **MNS** | 魔法数字 / 魔法字符串 | 10 | 未解释的字面量、反复出现的魔串等；每条须含：**位置**（文件与符号名）、**为何可疑**、**建议命名或配置化方向**。若本轮仅查此类，**不得**讨论其他坏味道类型。 |
| **LONGM** | 超长方法 | 8 | 各函数/方法：**行数**、**职责是否混杂**、是否含多段可独立语义块；拆分思路可到**命名级**。若本轮仅查此类，**不得**讨论类级别或分支结构。 |
| **LONGC** | 超长类 / 单文件堆积 | 6 | 哪些文件/类承担过多职责（如界面、数据访问、业务规则混同），或行数/成员异常集中；给出**候选拆分边界**（文字描述即可）。 |
| **BRANCH** | 多且深的分支 | 8 | 嵌套层级高、`switch`/`if` 链过长处；说明**认知负担**来源（缺少早退、缺少表驱动等）。 |
| **DUP** | 重复代码 | 6 组 | 重复或高度相似片段；每组：**出现位置**、**合并为单一实现**的建议接口形态（抽象层面）。 |
| **MIX** | 职责混杂 | 6 | 从职责边界审视：**层次错乱**或横切关注点堆叠（如视图层直接拼 SQL）；**问题陈述 + 建议边界**。 |
| **ERR** | 错误与边界被忽略 | 10 | 吞异常、过宽捕获、缺省分支缺失、输入未校验、超时/部分失败未覆盖等。 |
| **COUP** | 耦合过紧 / 隐式依赖 | 8 | 全局可变状态、单例滥用、跨层直连、隐式调用顺序等；说明**为何增改风险高**。 |
| **SEC** | 敏感信息与环境配置写死 | 逐项列出 | 疑似密钥、Token、密码、私网地址、云 AK/SK 等硬编码；**位置**与**敏感级**（高/中/低）。**禁止**在报告中回显完整秘值，须掩码处理。 |
| **DOC** | 注释 / 文档与实现脱节 | 10 | 对照注释、docstring、README 与**当前实现**；列出明显不一致处（引用具体段落或标题）。 |

**习惯建议**（与课程一致）：宜先**仅发现 → 人工确认 → 再修复**，避免单次扫描后自动大范围改写，以节约 **Token** 与时间。

## 输出格式（必选）

采用**两级列表**：

- **第一级**：问题类型，使用项目符号，并冠以 **代码 · 简短标题**，例如：`- **MNS · 魔法数字 / 魔法字符串**`
- **第二级**：该类型下的发现项，每条子项以 **类型代码-序号** 开头，且须包含：
  - **位置**：文件路径及符号名（若适用）
  - **为何可疑** 或 **问题陈述**
  - **建议方向**：命名、配置、拆分边界等（**不提供**具体补丁或改写后的完整代码，除非用户另行要求）

示例结构：

```markdown
## 代码坏味道发现（只读）

**分析范围**：`<用户指定的路径>`

- **MNS · 魔法数字 / 魔法字符串**
  - **MNS-1** …
  - **MNS-2** …
- **LONGM · 超长方法**
  - **LONGM-1** …
```

若用户要求**扫描全部类型**，按上表逐类检视并遵守各自上限；无问题的类型可写一条子项 **「未发现」**，或可省略该类型（须在回复中说明已检视且无发现）。

## 用户如何引用编号

在报告末尾用一两句话说明：用户可在后续对话中使用 **「修复 MNS-2、DUP-1」** 等形式，或粘贴已确认条目，供「仅修复」类提示词或另一 Skill 使用。**本 Skill 仅负责发现与编号，不执行修复。**

## 结语（可选用一句话）

坏味道并非语法错误，而是**可维护性与可靠性的早期信号**；后续若需借助模型改进，可要求其**减少魔法常量、缩短方法与类的职责范围、压平分支、抽取重复、补全错误分支、检查敏感信息是否硬编码**——但**本 Skill 的职责止于输出带编号的清单**。
